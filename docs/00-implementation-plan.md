# 프로젝트 구현 계획

이 문서는 PetOps Commerce를 어떤 의도와 순서로 설계하고 구현하는지 기록합니다. 포트폴리오를 보는 사람이 코드만 보는 것이 아니라, 문제 정의부터 아키텍처 결정, ERD, API, 구현, 검증까지의 흐름을 이해할 수 있도록 정리합니다.

## 1. 프로젝트 목표

PetOps Commerce의 목표는 PHP/MySQL 기반 실무 경험을 Java/Spring Boot 백엔드 포트폴리오로 자연스럽게 확장하는 것입니다.

기존 경험과 연결되는 부분은 주문, 재고, 운영 데이터, 관리자 기능, DB 설계입니다. 새롭게 증명할 부분은 Spring Boot, JPA, QueryDSL, Redis, Message Queue, Docker, 로드밸런싱, 모니터링, 테스트, AI/데이터 확장입니다.

최종적으로는 단순 CRUD 프로젝트가 아니라 실제 서비스 운영 관점에서 설계된 백엔드 시스템을 만드는 것을 목표로 합니다.

## 2. 채용 공고 분석 기반 설계 방향

최근 백엔드 채용 공고에서 반복적으로 등장한 요구사항을 프로젝트 기능과 기술 선택에 반영합니다.

| 채용 요구 | 프로젝트 반영 방식 |
|---|---|
| Java/Spring Boot 개발 경험 | 핵심 API 서버를 Java 21, Spring Boot 3.5.x로 구현 |
| JPA/SQL/트랜잭션 이해 | 주문, 재고, 쿠폰 도메인에서 JPA와 트랜잭션 경계 설계 |
| DB 설계 경험 | 회원, 상품, 주문, 재고, 결제, 쿠폰 ERD 직접 작성 |
| RESTful API 설계 | Swagger/OpenAPI 기반 API 명세 작성 |
| Redis/캐시/분산락 | 상품 캐시, 토큰 블랙리스트, 재고 차감 락 적용 |
| Message Queue | 주문 생성 후 재고/알림 이벤트 비동기 처리 |
| Docker/서버 분리 | Docker Compose로 DB, Redis, MQ, API 서버 분리 |
| 로드밸런싱 | Nginx 앞단에서 API 서버 3개로 요청 분산 |
| 모니터링/로그 | Actuator, Prometheus, Grafana로 상태 확인 |
| 성능 개선 | N+1 해결, 인덱스 적용, 부하 테스트 결과 문서화 |
| AI/데이터 활용 | 운영 데이터 요약, 고객 응대 초안 생성 기능으로 확장 |

## 5. 구현 원칙

- 먼저 작게 완성하고, 이후 운영 기술을 단계적으로 붙입니다.
- 처음부터 큰 MSA로 만들지 않고 모듈러 모놀리스로 시작합니다.
- 도메인 경계는 주문, 재고, 상품, 회원, 쿠폰, 운영 이벤트 기준으로 나눕니다.
- API 서버는 stateless하게 설계합니다.
- 사용자 세션은 서버 메모리에 두지 않고 JWT와 Redis를 활용합니다.
- 재고 차감처럼 위험한 로직은 동시성 테스트로 검증합니다.
- 기술을 많이 쓰는 것보다 각 기술을 왜 썼는지 문서로 설명합니다.

## 6. 학습하면서 구현하는 진행 방식

이 프로젝트는 코드만 빠르게 만드는 방식으로 진행하지 않습니다. 구현 단계마다 낯선 개념을 쉬운 말로 풀어 설명하고, 지금 만든 코드가 전체 아키텍처에서 어떤 역할을 하는지 함께 정리합니다.

각 구현 단계는 아래 형식으로 진행합니다.

| 항목 | 설명 |
|---|---|
| 지금 구현하는 것 | 이번 단계에서 실제로 만드는 기능이나 설정 |
| 왜 필요한가 | 실제 서비스 운영이나 채용 요구사항과 연결되는 이유 |
| 핵심 용어 | 모듈러 모놀리스, stateless, JPA, Redis 같은 용어를 쉬운 말로 설명 |
| 코드 구조 | 생성하거나 수정한 package, class, 설정 파일의 역할 |
| 구현 흐름 | 요청이 들어와서 controller, service, repository, DB까지 어떻게 흐르는지 설명 |
| 검증 방법 | 테스트, 빌드, API 호출, 로그 확인 등 확인 방법 |
| 포트폴리오 문장 | 면접이나 README에서 이 작업을 어떻게 설명하면 좋은지 정리 |
| 다음 단계 | 이번 작업이 다음 기능과 어떻게 이어지는지 정리 |

예를 들어 `API 서버는 stateless하게 설계한다`는 문장을 그냥 원칙으로만 두지 않습니다. 인증 기능을 구현할 때 JWT를 왜 쓰는지, 서버 메모리에 로그인 세션을 저장하면 로드밸런싱에서 어떤 문제가 생기는지, Redis는 어떤 경우에 필요한지 함께 설명합니다.

또 `모듈러 모놀리스`도 추상적인 용어로 끝내지 않습니다. 실제 package를 `member`, `product`, `order`, `stock`, `coupon`처럼 나누면서, 왜 이 경계가 나중에 서버 분리나 MSA 전환의 출발점이 되는지 같이 확인합니다.
## 7. 포트폴리오 강조 메시지

이 프로젝트에서 강조할 핵심 메시지는 다음과 같습니다.

- PHP/MySQL 실무 경험을 Java/Spring 기반 시스템 설계로 확장했습니다.
- 주문, 재고, 쿠폰, 운영 이벤트처럼 실제 서비스에 가까운 도메인을 설계했습니다.
- JPA, 트랜잭션, 인덱스, 동시성 제어를 기능 안에서 다뤘습니다.
- Docker Compose와 Nginx로 다중 API 서버와 로드밸런싱을 구성했습니다.
- Redis, Message Queue, Batch, Monitoring을 운영 관점에서 단계적으로 적용했습니다.
- 모든 구현 과정을 문서화해 문제 정의, 설계, 구현, 검증 흐름을 보여줍니다.
