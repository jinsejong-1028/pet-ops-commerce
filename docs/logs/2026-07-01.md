- 판매 주문 확정 시 `sales_orders.warehouse_id` 필수 검증을 추가했습니다.

- `customer_order_items.status`를 추가하고 확정/취소 시 `CONFIRMED`/`CANCELED`로 변경합니다.
- `OrderItem`을 `BaseAuditEntity` 상속 구조로 바꿔 `updated_by`가 자동 기록되도록 정리했습니다.

- `docs/api/01-api-spec.md`, `docs/architecture/02-erd.md`, `docs/domains/06-order-domain.md`, `http/order-api.http`를 최신화했습니다.

### Migration 기준

- 기존 migration은 수정하지 않고 `V6__add_sales_order_warehouse_and_customer_order_status.sql`을 추가했습니다.
- 기존 데이터 보정 `update/delete/임시 insert`는 migration에 넣지 않았습니다.
- `customer_order_items.status`는 기존 row를 고려해 nullable로 추가하고 신규 데이터부터 애플리케이션에서 `CREATED`를 저장합니다.

### 확인 결과

- 사용자가 `bootRun`을 실행했고, V6 checksum mismatch가 발생했습니다.
- 원인은 이미 DB에 적용된 V6 migration 파일에 주석을 추가해 checksum이 바뀐 것이었습니다.
- V6 파일을 적용 당시 내용으로 되돌려 checksum mismatch 원인을 제거했습니다.
- 이후 사용자가 서버 동작을 확인했습니다.

## 반복 방지 메모

- 이미 DB에 적용된 Flyway migration은 주석만 바꿔도 checksum mismatch가 발생합니다.
- PostgreSQL은 `ALTER TABLE ADD COLUMN`에서 물리 컬럼 위치를 중간으로 지정할 수 없습니다.
- 컬럼 물리 순서는 새 `create table` migration 작성 시점에 맞추고, 기존 테이블의 추가 컬럼은 Entity/문서의 논리 순서로 관리합니다.
